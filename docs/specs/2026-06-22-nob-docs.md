# Feature: nob:docs — Inline Documentation Agent

## Summary
A pipeline skill and standalone agent that reads changed files after a dev run and writes or updates stack-appropriate inline documentation (JSDoc, docstrings, GoDoc, etc.) so docs never drift from what was actually implemented.

## Users
Developers using nob to implement features, invoked either automatically by the hub after a dev run or standalone via `/nob:docs` pointed at a file or unit.

## User flow

### Hub-dispatched (automatic)
1. Hub completes the dev phase and has a `[DEV OUTPUT]` block listing every changed and created file.
2. Hub dispatches the docs agent with the dev output, units list, and working directory.
3. Docs agent reads each changed file and identifies public exports, functions, methods, and classes that are missing inline docs or have stale docs (signature changed since the doc was written).
4. For each gap, docs agent writes a stack-appropriate doc block (JSDoc, Google-style docstring, GoDoc comment, etc.) directly above the symbol.
5. Docs agent emits a `[DOCS OUTPUT]` block listing every file touched, how many docs were added vs. updated, and files skipped because they were already fully documented.
6. Hub passes the docs output to the reviewer along with dev output.

### Standalone
1. Developer runs `/nob:docs` with a file path, unit name, or spec file as the target.
2. Docs agent reads the target file(s) for the named unit (or all files in the unit if only a unit name is given).
3. Same analysis and write steps as above.
4. Emits `[DOCS OUTPUT]` and stops — does not invoke reviewer.

## Requirements
- Read `[DEV OUTPUT]` (hub-dispatched) or user-specified paths (standalone) to determine which files to document.
- Detect the stack type per unit from `.nob.yml` or by sniffing the file extension/manifest; use it to choose the correct doc format.
- Supported doc formats: JSDoc (`/** */`) for JS/TS, Google-style docstrings for Python, GoDoc line comments for Go, Javadoc for Java, KDoc for Kotlin, DartDoc for Dart/Flutter.
- Document only public-facing symbols: exported functions, public methods, public classes, exported constants. Skip private/internal helpers.
- Preserve existing correct doc blocks — do not overwrite docs where the signature is unchanged.
- Update (not replace) a doc block when a function's signature changed since the block was written; preserve any correct `@param` / `@returns` lines and only fix the outdated ones.
- Never touch business logic — only add or update comment blocks.
- Emit a `[DOCS OUTPUT]` block (opening and closing tags required) listing: files documented (with add/update counts), files skipped (already fully documented), and files that could not be processed.
- In the hub pipeline, run between dev and reviewer. The hub passes docs output to the reviewer so it can verify doc coverage as part of the acceptance check.
- Configurable: `agents.enabled.docs: false` in `.nob.yml` disables the automatic pipeline step (docs agent can still be run standalone).
- Reuse `skills/dev/stacks/` for per-stack conventions and doc style guidance.

## Acceptance criteria
- [ ] A JS/TS file with undocumented exported functions gets a JSDoc block added above each undocumented export.
- [ ] A Python file with undocumented public functions gets a Google-style docstring added inside each undocumented function body.
- [ ] A Go file with undocumented exported identifiers gets a GoDoc comment line added above each.
- [ ] Existing correct doc blocks are left unchanged (verified by a file checksum or line-diff check).
- [ ] A function whose signature changed since its existing doc block gets its `@param`/`@returns` lines updated, not the entire block replaced.
- [ ] Private/unexported symbols are not documented.
- [ ] `[DOCS OUTPUT]` lists every file modified with an add count and update count.
- [ ] `[DOCS OUTPUT]` lists files skipped with reason `already documented`.
- [ ] Running standalone with a file path produces `[DOCS OUTPUT]` and stops without invoking reviewer or any other pipeline step.
- [ ] `agents.enabled.docs: false` in `.nob.yml` causes the hub to skip dispatching the docs agent (pipeline proceeds directly from dev to reviewer).
- [ ] A file listed in `[DEV OUTPUT]` that cannot be read is reported in `[DOCS OUTPUT]` as skipped with reason `unreadable` — the agent does not crash.

## Builds on
- `skills/dev/stacks/` — per-stack implementation helpers (same stacks, same context files)
- `skills/dev/SKILL.md` — `[DEV OUTPUT]` block format that docs agent reads as input
- `skills/reviewer/SKILL.md` — reviewer receives `[DOCS OUTPUT]` alongside dev output; it can verify doc coverage as an acceptance criterion
- `skills/nob/SKILL.md` — hub integration point: docs agent dispatched between dev and reviewer

## Constraints
- No new runtime dependencies for the plugin.
- Must not modify any file not listed in `[DEV OUTPUT]` when hub-dispatched (scope enforcement mirrors the unit-boundary hook).
- Does not generate a documentation site (Docusaurus, MkDocs, Sphinx) — inline docs only.

## Error states
- File listed in DEV OUTPUT is unreadable: skip it, note `unreadable` in `[DOCS OUTPUT]`, continue.
- Stack type cannot be determined: emit a warning in `[DOCS OUTPUT]`, skip inline doc generation for that file, suggest the user add the unit's `type` to `.nob.yml`.
- Existing doc block is malformed (e.g. unclosed `/**`): report it in `[DOCS OUTPUT]` as `malformed — skipped`, do not overwrite.

## Out of scope
- Generating a documentation website or static site.
- Translating or internationalizing doc strings.
- Writing usage examples, tutorials, or README sections.
- Documenting private, unexported, or internal symbols.
- Linting prose quality or spelling inside existing doc blocks.
