# Feature: TDD Mode (`--tdd` flag)

## Summary
A `--tdd` flag for `/nob implement` that inserts a test-writing agent between Tech Lead and Dev, enforcing a Red-Green-Refactor cycle so tests are written before implementation.

## Users
Developers running `/nob implement` who follow test-driven development or want to ensure test coverage is written before code, not after.

## User flow
1. Developer runs `/nob implement docs/specs/my-feature.md --tdd`.
2. Hub detects `--tdd` flag, routes to lite or full path as normal.
3. Tech Lead completes its task list and contracts as usual.
4. **Red phase**: a new `nob:test-writer` agent reads the TL task list and spec acceptance criteria, writes failing tests for each task (one test file per unit), and emits `[TEST WRITER OUTPUT]` listing test files created.
5. Hub (or path skill) pauses and prints the generated test files. Prompts: `"Tests written. Review them, then continue? (yes / edit / skip-tdd)"`.
6. If `yes`: Dev runs against the existing codebase — tests should fail. Dev implements to make them pass (**Green phase**).
7. Dev emits `[DEV OUTPUT]` as normal, now including test results.
8. Reviewer verifies all tests pass as part of its acceptance check (**Refactor/verify phase**).
9. If `edit`: worktree is left open; user edits tests manually and types `continue` to resume.
10. If `skip-tdd`: TDD phase is skipped; run continues as a normal (non-TDD) implement.

## Requirements
- `--tdd` flag is parsed by the hub at Step 2 and passed via `[INPUTS]` to the path skill.
- A new `skills/test-writer/SKILL.md` agent: reads `[TECH LEAD OUTPUT]` task list + spec acceptance criteria, writes failing tests for each task, emits `[TEST WRITER OUTPUT]`.
- `[TEST WRITER OUTPUT]` required fields: `Units tested:`, `Test files written:`, `Tests written:`, `Framework detected:`.
- Test writer detects the test framework per unit from the stack type and installed dependencies (jest/vitest for node/next/react, pytest for python, `go test` for go, rspec for ruby).
- Test writer writes tests that are syntactically valid but intentionally fail (testing behavior not yet implemented).
- Dev agent, when `--tdd` is active, runs the test suite before implementing and confirms tests fail, then implements to make them pass.
- Reviewer's acceptance check treats failing tests as a FAIL when `--tdd` was active.
- `agents.enabled` list gains an optional `test-writer` entry; defaults to enabled when `--tdd` is passed and the agent is not explicitly disabled.
- `--tdd` is supported on lite and full paths only — quick path prints `"--tdd not supported on quick path"` and proceeds without it.
- `agents.models.test-writer` configurable in `.nob.yml`; default: `haiku`.

## Acceptance criteria
- [ ] Running `/nob implement spec.md --tdd` on a full-path run dispatches the test-writer agent after Tech Lead and before Dev.
- [ ] `[TEST WRITER OUTPUT]` is emitted with all required fields.
- [ ] Test files are written to the correct unit paths.
- [ ] Dev runs the test suite before implementing and confirms failures are present (logged in DEV OUTPUT).
- [ ] After Dev completes, all written tests pass.
- [ ] Reviewer marks the run FAIL if any `--tdd`-generated test still fails after Dev.
- [ ] `--tdd` on a quick-path run prints the unsupported message and continues without the test-writer phase.
- [ ] `skip-tdd` at the review prompt skips the TDD phase without aborting the run.
- [ ] `agents.enabled` with `test-writer` removed disables the test-writer even when `--tdd` is passed.
- [ ] Terminal summary includes a `TDD:` line showing `Red ✓ → Green ✓` or `skipped`.

## Builds on
- `skills/nob/SKILL.md` — hub flag detection and `[INPUTS]` passing
- `skills/nob/path-full/SKILL.md` — dispatch order (insert test-writer between TL and Dev)
- `skills/nob/path-lite/SKILL.md` — same insertion point for lite path
- `skills/tech-lead/SKILL.md` — `[TECH LEAD OUTPUT]` task list is the test-writer's primary input
- `skills/dev/SKILL.md` — Dev receives test files as context, runs tests before and after
- `skills/reviewer/SKILL.md` — Reviewer gains a TDD-pass check when flag was active
- `skills/dev/stacks/` — per-stack test framework detection

## Constraints
- No new runtime dependencies for the plugin.
- Test writer must not modify any existing test files — it only creates new ones.
- Test writer scope is bounded by the unit-boundary hook (same as Dev).

## Error states
- Test framework cannot be detected: emit warning in `[TEST WRITER OUTPUT]`, skip test generation for that unit, note `framework-unknown` in terminal summary.
- Test files cannot be written (permission error): abort TDD phase, prompt user to check permissions, offer `skip-tdd`.
- Dev cannot run tests (no test runner found): log `test-runner-not-found` in DEV OUTPUT, continue implementation, Reviewer skips TDD-pass check.

## Out of scope
- Mutation testing or test quality scoring.
- Generating tests for existing (pre-run) code — test writer only covers tasks in the current TL task list.
- Enforcing TDD on the quick path.

## Open questions
- Should `--tdd` be opt-in per run or configurable as always-on via `.nob.yml`? Defer to Tech Lead.
- When tests pass on first Dev attempt (no retry needed), should the Reviewer still re-run the suite independently? Defer to Tech Lead.
