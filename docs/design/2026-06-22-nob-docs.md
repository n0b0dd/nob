## Interfaces / contracts

### `[DOCS OUTPUT]` block (docs agent → hub → reviewer)

Produced by `skills/docs/SKILL.md`. Forwarded by the hub to the reviewer alongside `[DEV OUTPUT]`.

Shape:
```
[DOCS OUTPUT]
Files documented:
  - [path]: added N, updated M
Files skipped:
  - [path]: already documented
  - [path]: unreadable
  - [path]: malformed — skipped
  - [path]: stack type unknown — suggest adding `type` to .nob.yml for this unit
Total: added N doc blocks, updated M doc blocks across K files
[/DOCS OUTPUT]
```

Required fields (validated by hub's Output Block Validation table):
- `Files documented:` (list, or: `none`)
- `Files skipped:` (list, or: `none`)
- `Total:` (summary line)

### Hub dispatch contract (hub → docs agent)

`[INPUTS]` block passed to `skills/docs/SKILL.md`:
- `Working directory:` — project root path
- `Dev output:` — full `[DEV OUTPUT]` block text (hub-dispatched); used to extract changed/created file list
- `Units:` — one line per unit: `  - name: {name}, type: {type}, path: {path}` (for stack-type detection)
- `.nob.yml contents:` — raw text (for `agents.enabled` check in standalone mode; passed as-is)
- `CLAUDE.md contents:` — raw text (for project conventions)
- `Project memory:` — structured project memory summary string
- `Standalone target:` — file path or unit name when invoked via `/nob:docs <target>`; value `none` when hub-dispatched

### Reviewer doc-coverage check contract

When `[DOCS OUTPUT]` is present in context, reviewer adds a doc-coverage section:
- For each file in `Files documented:`, verify the documented counts match expectations.
- If `Files skipped:` contains lines with reason `unreadable` or `malformed — skipped`, surface them as NEEDS REVIEW items.
- If `Files skipped:` is non-empty with reason `stack type unknown`, add to Items for human review.
- Missing docs (undocumented public symbols not reported as added) → NEEDS REVIEW (never FAIL).
- Doc coverage check is SKIPPED when `[DOCS OUTPUT]` is absent from context.

## Task list

- id: t1
  title: implement skills/docs/SKILL.md (docs agent skill)
  description: >
    Create `skills/docs/SKILL.md` — the self-contained instruction set for the nob:docs inline documentation agent.
    The skill must:
    - Detect mode (hub-dispatched vs. standalone) via presence of `[INPUTS]` block.
    - Hub-dispatched: read `Dev output:` from `[INPUTS]` to get the file list; standalone: read `Standalone target:` (file path or unit name).
    - For each file, detect stack type from `Units:` in `[INPUTS]` or by sniffing extension/manifest.
    - Stack-to-format mapping: JS/TS (.js, .ts, .jsx, .tsx) → JSDoc (`/** */`); Python (.py) → Google-style docstring; Go (.go) → GoDoc comment; Java (.java) → Javadoc; Kotlin (.kt) → KDoc; Dart (.dart) → DartDoc.
    - Per file: read it, identify public-facing symbols (exported functions, public methods, classes, constants — skip private/internal).
    - For each undocumented symbol: add the appropriate doc block above (JS/TS/Java/Kotlin/Go/Dart) or inside (Python) the symbol.
    - For each symbol with an existing doc block: compare the current signature against the doc block. If signature changed, update only the outdated `@param`/`@returns` lines; preserve the rest.
    - For each symbol with an existing correct doc block: leave it unchanged.
    - Never touch business logic — only add/update comment blocks.
    - Error states: unreadable file → skip + note `unreadable`; undetermined stack type → skip + note `stack type unknown`; malformed existing doc block (e.g. unclosed `/**`) → skip that block + note `malformed — skipped`.
    - Standalone mode: emit `[DOCS OUTPUT]` and stop; do NOT invoke reviewer or any other pipeline step.
    - Emit `[DOCS OUTPUT]` block with required fields: `Files documented:`, `Files skipped:`, `Total:`.
    - Include frontmatter: `name: docs`, `description: ...`, and a trigger description matching `/nob:docs`.
    - Scope enforcement (hub-dispatched): only document files listed in `[DEV OUTPUT]`; do not wander outside that list.
    - Reuse `skills/dev/stacks/` for per-stack conventions: instruct the agent to read the relevant stacks file for doc-style guidance.
  unit: root
  files: skills/docs/SKILL.md
  depends_on: empty

- id: t3
  title: update reviewer to handle [DOCS OUTPUT] (skills/reviewer/SKILL.md)
  description: >
    Modify `skills/reviewer/SKILL.md` to read and report on `[DOCS OUTPUT]` when present.
    Changes required:
    1. **Step 3 — Read implementation outputs:** Add instructions to extract `[DOCS OUTPUT]` from context. Store as DOCS_OUTPUT (or `none` if absent).
    2. **New Step 3.9 — Doc coverage check** (insert after Step 3.8 Design compliance):
       - If DOCS_OUTPUT is absent or `none`: set DOC_COVERAGE_STATUS = "SKIPPED — no docs output". Skip to Step 4.
       - Extract `Files skipped:` from DOCS_OUTPUT. For each entry:
         - Reason `malformed — skipped`: add to Items for human review: `Docs: malformed doc block in {path} — skipped by docs agent, review manually`.
         - Reason `stack type unknown`: add to Items for human review: `Docs: stack type could not be detected for {path} — add unit type to .nob.yml`.
         - Reason `unreadable`: informational — do NOT add to Items for human review (file was likely deleted; the dev agent already noted it).
         - Reason `already documented`: no action.
       - If any IMPORTANT doc items were added to Items for human review: set DOC_COVERAGE_STATUS = "NEEDS REVIEW — {N} item(s)". Overall status cannot be PASS.
       - Otherwise: set DOC_COVERAGE_STATUS = "PASS" (or "SKIPPED" if DOCS_OUTPUT was none).
    3. **Output block:** Add `Doc coverage:` field after `Design compliance:`:
       ```
       Doc coverage:
         Status: [PASS | NEEDS REVIEW: N item(s) | SKIPPED — no docs output]
         [if NEEDS REVIEW: list each item as "- [IMPORTANT] {path} | {reason}"]
       ```
    4. **Output Block Validation table in hub:** already handled by t2; reviewer just needs to emit the field.
  unit: root
  files: skills/reviewer/SKILL.md
  depends_on: t2
